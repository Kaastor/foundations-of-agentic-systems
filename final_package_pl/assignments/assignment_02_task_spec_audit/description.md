Spotkanie 2: „Czym dokładnie jest zadanie?”

**Po co istnieje to zadanie:** Studenci mają nauczyć się, że prośba użytkownika nie jest jeszcze kontraktem zadania gotowym do sterowania systemem.

**Główny akcent pętli rozumowania:** `wygeneruj -> porównaj -> popraw -> uzasadnij`

**Oczekiwany czas:** `4 godziny`

**Poziom kodowania:** `ograniczone czytanie kodu`, z opcjonalnym rerunem workflow

## Zadanie

Użyj niejednoznacznej prośby o przegląd literatury, aby zrobić trzy rzeczy:

1. przygotować albo uzyskać dwie sensowne kandydackie specyfikacje zadania,
2. porównać je z zapisanymi specyfikacjami kursowymi,
3. poprawić jedną finalną, ograniczoną specyfikację zadania,
4. obejrzeć mały fragment implementacji pokazujący, jak prośba staje się zweryfikowanym kontraktem zadania.

Końcowy raport powinien obronić:

- które pola są niezbędne dla sterowania,
- dlaczego jeden kandydat jest mocniejszy,
- jaka niejednoznaczność pozostaje i jak należy ją obsłużyć.

## Checklista oddania

- uwzględnij co najmniej dwa kierunki kandydackich specyfikacji zadania,
- porównaj je z co najmniej dwiema zapisanymi specyfikacjami,
- omów cele, ograniczenia, sukces oraz zatrzymanie lub przekazanie,
- obejrzyj co najmniej jeden nazwany moduł źródłowy i wyjaśnij, które pola czyni on operacyjnie koniecznymi,
- podaj jedną poprawioną finalną specyfikację zadania,
- wskaż jedną niejednoznaczność, którą zostawiłbyś jawnie zamiast zgadywać.
